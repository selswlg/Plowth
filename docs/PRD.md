# PRD 상세 초안

## 1. 개요

### 한 줄 정의

**AI가 학습 자료를 자동으로 구조화하고, 개인별 기억 곡선과 실수 패턴을 반영해 복습·이해·약점 보완까지 수행하는 학습 운영체제형 앱**

조금 더 제품적으로 표현하면:

**“설정 없는 Anki, 얕지 않은 Duolingo, 그리고 학습 맥락을 이해하는 AI 튜터를 하나로 합친 도구.”**

---

### 제품 목표

이 제품의 목표는 단순히 “플래시카드 앱”을 만드는 것이 아니다.
핵심 목표는 아래 5가지다.

1. **학습 진입 장벽 제거**
   사용자가 덱, 태그, 스케줄링 규칙, 카드 설계 방식을 공부하지 않아도 바로 학습에 들어갈 수 있어야 한다.

2. **고급 학습 기능의 자동화**
   Anki에서 Add-on, 설정, 시행착오로 해결하던 기능을 제품 기본 기능으로 흡수한다.
   예: 자동 카드 생성, 자동 스케줄링, 약점 분석, 진행 시각화.

3. **암기에서 이해까지 확장**
   단순 정답 회상만이 아니라 “왜 틀렸는지”, “관련 개념이 무엇인지”, “어떤 맥락에서 쓰이는지”까지 학습 경험에 포함한다.

4. **개인화된 학습 루프 구축**
   사용자별 반응 속도, 정오답, 반복 실수 패턴, 학습 시간대를 분석해서 복습 간격과 설명 방식을 조정한다.

5. **신뢰 가능한 학습 도구 제공**
   속도, 오프라인 대응, 동기화 안정성, 데이터 보존성에서 파워 유저가 믿고 쓸 수 있어야 한다.

---

### 배경

현재 학습/자기계발 앱 시장은 크게 세 가지 결핍을 보인다.

첫째, **강력하지만 어렵다.**
Anki는 spaced repetition, Add-on, 커스터마이징 측면에서 강력하지만, 초보자에게는 덱 구조, 카드 설계, 스케줄링 옵션, Add-on 의존성이 큰 장벽이 된다.

둘째, **쉽지만 얕다.**
Duolingo류 앱은 접근성이 좋고 습관 형성에 유리하지만, 반복형 문제 풀이가 많고 문법 설명, 실제 적용 맥락, 깊이 있는 이해 지원이 부족하다는 불만이 많다.

셋째, **콘텐츠는 있어도 개인화된 학습 운영이 약하다.**
Khan Academy, Coursera 등은 콘텐츠 전달에는 강점이 있지만, 개인의 기억 상태나 약점 패턴에 맞춰 학습 구조 자체를 조정하는 수준은 제한적이다.

이 제품은 이 틈을 겨냥한다.

* **도구의 강력함은 유지**
* **설정 부담은 제거**
* **AI를 ‘채팅 기능’이 아니라 학습 운영 엔진으로 사용**
* **기억, 이해, 분석, 동기 시스템을 하나의 파이프라인으로 통합**

## 2. 핵심 가치

### 사용자 문제

#### 문제 1. 학습 도구를 배우느라 학습이 늦어진다

기존 고급 학습 도구는 사용법 자체가 어렵다.
유저는 공부를 시작하기 전에 덱 구조, 템플릿, 태그, 알고리즘, Add-on 설치 등을 먼저 익혀야 한다.

#### 문제 2. 카드 생성과 관리가 너무 수동적이다

좋은 복습 시스템을 쓰려면 결국 카드 품질이 중요하다.
하지만 많은 유저가 직접 카드를 만들고, 편집하고, 분류하는 과정에서 피로를 느낀다.
특히 긴 텍스트나 PDF를 학습 자료로 쓸 때 이 마찰이 매우 크다.

#### 문제 3. 틀려도 왜 틀렸는지 구조적으로 알기 어렵다

대부분의 앱은 “맞았다/틀렸다” 정도만 기록한다.
하지만 실제 학습에서는 “개념 자체를 모르는 건지”, “비슷한 개념과 헷갈린 건지”, “시제나 조건처럼 반복되는 실수인지”가 중요하다.

#### 문제 4. 학습이 기억 중심에만 머문다

암기 앱은 회상은 잘 시키지만 개념 이해, 맥락 연결, 실제 활용 예시 제공은 부족한 경우가 많다.
반대로 학습 콘텐츠 앱은 설명은 많지만 반복 최적화는 약하다.

#### 문제 5. 파워 유저는 강력함을 원하지만, 일반 유저는 단순함을 원한다

시장에 있는 제품은 대체로 둘 중 하나만 만족시킨다.
강하면 복잡하고, 쉬우면 얕아진다.
이 모순이 아직 완전히 해결되지 않았다.

### 해결 방식

#### 1. Zero Setup 학습 구조

사용자는 자료만 넣는다.
텍스트, PDF, 링크, 직접 입력한 노트를 넣으면 AI가 이를 분석해 핵심 개념을 뽑고, atomic card로 쪼개고, 학습 가능한 카드 세트로 구성한다.

#### 2. AI 기반 카드 생성 및 정제

AI가 다음을 수행한다.

* 핵심 개념 추출
* 개념 간 선후 관계 파악
* 카드 유형 분류

  * 정의형
  * 원리형
  * 비교형
  * 적용형
* 과도하게 넓은 카드 분해
* 모호하거나 중복되는 카드 제거 제안

즉, 사용자는 카드 제작자가 아니라 **학습 감독자**가 된다.

#### 3. 개인화된 복습 엔진

기본 복습 엔진은 FSRS 계열의 기억 모델을 기반으로 하되, 다음 입력까지 반영한다.

* 정답/오답
* 응답 속도
* 연속 실패 여부
* 관련 개념군의 실패
* 사용자의 시간대별 성과
* 최근 피로도 추정

이를 통해 단순 카드 단위가 아니라 **개념 네트워크 단위**로 복습 중요도를 조정한다.

#### 4. 오답 중심 학습 루프

사용자가 오답을 내면 시스템은 단순히 다음 복습 일정을 잡는 데서 끝나지 않는다.

* 왜 틀렸는지 짧게 설명
* 유사 개념과의 차이 제시
* 예문/예시 제공
* 비슷한 유형의 추가 카드 제공
* 오답 패턴 테이블 누적

이를 통해 “틀린 기록”이 아니라 **학습 진단 데이터**로 전환된다.

#### 5. AI 코치형 개입

AI는 항상 전면에 나오지 않는다.
필요한 순간에만 개입한다.

예:

* “이 개념을 3번 연속 헷갈리고 있어. 핵심 차이만 짚어볼까?”
* “지금 복습하면 기억 유지 확률이 가장 높아.”
* “너는 밤 시간대에 정확도가 떨어지는 편이야.”

즉, AI는 채팅 기능이 아니라 **학습 흐름을 보정하는 코치 레이어**다.

### 차별점

#### 차별점 1. Add-on 의존성을 제품 내부 기능으로 흡수

Anki의 강점은 Add-on 생태계이지만, 동시에 복잡성의 원인이기도 하다.
이 제품은 Review Heatmap, 자동 스케줄링, 이미지 기반 학습, 고급 검색, 카드 생성 보조 같은 니즈를 외부 Add-on이 아닌 기본 기능으로 제공한다.

#### 차별점 2. 기억 + 이해 + 분석의 통합

대부분의 앱은 아래 셋 중 하나에 치우친다.

* 기억 최적화
* 설명 중심 학습
* 동기 부여

이 제품은 세 가지를 하나의 루프로 묶는다.

**입력 → 카드 생성 → 복습 → 오답 분석 → 설명 → 재학습 → 스케줄 조정**

#### 차별점 3. “보이지 않는 복잡성”

UI는 단순하게 유지하되, 내부 로직은 매우 강력하게 설계한다.
사용자는 쉽게 쓰지만, 내부에서는 AI 오케스트레이션, 기억 모델, 실수 패턴 추적, 연관 개념 그래프가 작동한다.

#### 차별점 4. 파워 유저 친화적 신뢰성

이 제품은 단순한 캐주얼 학습 앱이 아니다.
오프라인 학습, 빠른 리뷰 전환, 데이터 내보내기, 안정적 동기화, 낮은 지연시간을 핵심 가치로 둔다.

#### 차별점 5. “정답 맞히기”가 아니라 “왜 헷갈리는지”를 다룬다

학습 진전은 단순 회상률보다 오답 패턴과 약점 구조를 얼마나 잘 파악하느냐에 달려 있다.
이 제품은 실패 데이터를 중심으로 개인 학습 모델을 구축한다.

## 3. 타겟 및 시나리오

### 타겟 사용자

#### 1차 타겟: 고강도 학습자 / 파워 유저

가장 먼저 공략해야 할 사용자군이다.

대표 특성:

* 이미 Anki나 유사 도구를 써봤다
* 시험, 자격증, 언어, 전문지식 학습에서 효율에 민감하다
* 카드 품질, 복습 알고리즘, 속도, 데이터 신뢰성을 중요하게 생각한다
* 복잡한 도구도 감수할 수 있지만, 더 쉬운 상위호환이 나오면 이동할 가능성이 높다

예:

* 의대/치대/약대/간호 계열 학생
* 공무원/전문 자격증 수험생
* GRE, TOEFL, JLPT, TOPIK, SAT, USMLE 등 장기 학습자
* 개발/법률/재무 등 전문 용어 암기와 개념 이해가 필요한 직군

#### 2차 타겟: 자기주도 학습자

* 노트, 블로그, 강의 자료를 스스로 정리해 공부하는 사용자
* Notion, Obsidian, Docs 등에 흩어진 자료를 학습용으로 전환하고 싶은 사용자

#### 3차 타겟: 언어 학습자

* Duolingo식 가벼운 진입은 좋지만, 실제로는 더 깊은 학습이 필요한 사용자
* 단어 암기 + 문법 + 문맥 적용을 통합하고 싶은 사용자

---

### 사용 시나리오

#### 시나리오 1. 시험 준비생의 노트 기반 학습

1. 사용자가 생물학 요약 노트를 앱에 붙여넣는다.
2. 앱이 핵심 개념을 추출한다.
3. 개념별로 atomic card를 생성한다.
4. 사용자는 바로 학습을 시작한다.
5. 오답이 반복되는 개념은 AI가 짧은 설명과 유사 문제를 제공한다.
6. 시험일까지 기억 유지율이 높은 상태로 반복 학습한다.

#### 시나리오 2. 언어 학습자의 문맥 기반 복습

1. 사용자가 문장 묶음이나 기사 일부를 넣는다.
2. 앱은 단어 카드만 만드는 것이 아니라 시제, 표현 차이, 문법 포인트를 포함한 카드 세트를 만든다.
3. 복습 도중 틀리면 “왜 이 시제가 쓰였는지”, “비슷한 표현과 차이가 무엇인지”를 설명한다.
4. 사용자는 단순 단어 암기가 아닌 실제 문맥 이해를 함께 쌓는다.

#### 시나리오 3. 자기계발형 사용자의 학습 운영 자동화

1. 사용자는 여러 자료를 저장해둔다.
2. 앱은 이를 개념별로 정리하고, 주간 학습 큐를 자동 구성한다.
3. 사용자는 매일 “오늘의 복습”만 수행하면 된다.
4. 인사이트 화면에서 약점 영역, 누적 성장, 학습 시간대 성과를 확인한다.

#### 시나리오 4. Anki 이탈 유저의 전환

1. 사용자는 더 쉬운 도구를 찾고 있다.
2. 앱은 기존 학습 흐름보다 빠른 입력, 자동 카드 생성, 쉬운 리뷰 인터페이스를 제공한다.
3. 사용자는 설정 없이 시작하면서도 고급 복습 정확도를 경험한다.
4. 데이터 export/import, 빠른 속도, 안정적 동기화 덕분에 도구 신뢰를 형성한다.

## 4. 성공 지표

### 핵심 KPI

#### 제품 사용성 KPI

1. **첫 학습 시작 시간**

   * 목표: 앱 설치 후 첫 카드 리뷰까지 3분 이하
   * 의미: 온보딩 마찰을 줄였는지 측정

2. **자료 입력 → 카드 생성 완료율**

   * 목표: 입력한 사용자의 70% 이상이 카드 생성까지 완료
   * 의미: 입력 UX와 생성 품질의 적합성 확인

3. **첫 세션 내 리뷰 완료 수**

   * 목표: 첫 세션에서 최소 10카드 이상 리뷰하는 사용자 비율 60% 이상
   * 의미: 진입 후 즉시 가치 체감 여부

#### 유지/리텐션 KPI

4. **D1, D7, D30 리텐션**

   * 목표 예시:

     * D1 > 45%
     * D7 > 30%
     * D30 > 15%
   * 초반에는 D7이 가장 중요하다.

5. **주간 리뷰 수행률**

   * 목표: 활성 사용자 중 주 4회 이상 리뷰 비율 50% 이상

6. **활성 사용자당 평균 리뷰 카드 수**

   * 목표: DAU당 평균 리뷰 카드 20장 이상

#### 학습 품질 KPI

7. **오답률 개선 추이**

   * 동일 개념군에서 2주 단위 오답률 감소 여부 측정

8. **기억 유지율**

   * 복습 주기 이후 회상 성공률 추적

9. **AI 설명 사용 후 정답률 개선**

   * Why 버튼 사용 전후 같은 유형 카드의 정답률 변화를 측정

#### 전환/수익 KPI

10. **Free → Paid 전환율**

* 목표: 고빈도 유저 중심으로 3~8% 초기 목표

11. **유료 사용자 월간 유지율**

* 파워 유저가 실제 도구 가치에 돈을 지불하는지 확인

### 리스크/이슈

#### 1. AI 비용이 빠르게 증가할 수 있다

카드 생성, 설명, 분석, 연관 개념 추출까지 모두 실시간 호출하면 비용이 높아진다.

대응:

* 캐싱
* 동일 카드 설명 재사용
* 긴 입력은 비동기 처리
* 무료 플랜에서 AI 호출 한도 관리
* 중요 기능만 고성능 모델, 나머지는 경량 모델 사용

#### 2. AI 카드 품질이 일관되지 않을 수 있다

카드가 너무 넓거나, 모호하거나, 시험형으로 적합하지 않을 수 있다.

대응:

* strict JSON 출력
* 후처리 검증
* atomicity 검사
* 사용자 수정 기능
* 인기 수정 패턴을 재학습 룰에 반영

#### 3. 파워 유저가 “장난감 같다”고 느낄 수 있다

너무 단순한 UI는 오히려 신뢰를 떨어뜨릴 수 있다.

대응:

* 기본 UI는 간단하게, 고급 인사이트와 설정은 숨겨진 전문가 모드로 제공
* 데이터 export/import 지원
* 리뷰 속도 최적화
* 알고리즘 투명성 일부 공개

#### 4. 동기화/데이터 손실 문제 발생 시 신뢰가 무너진다

학습 데이터는 매우 민감하다.

대응:

* 오프라인 로그 저장
* append-only review log
* 자동 백업
* 충돌 복구 정책
* 변경 내역 추적

#### 5. 너무 많은 기능을 넣으면 초기 제품이 느슨해질 수 있다

초기에는 “모든 학습 앱을 대체”하려 하면 안 된다.

대응:

* MVP 범위 엄격 관리
* 핵심 루프 우선
* Capture → Generate → Review → Explain → Insight 중심으로 출시

## 7. 구현 방법(디테일하게)

### 전체 아키텍처

권장 구조는 다음과 같다.

* **클라이언트:** Flutter 또는 React Native
* **웹/관리자:** React / Next.js
* **백엔드 API:** FastAPI
* **DB:** PostgreSQL
* **캐시:** Redis
* **비동기 작업:** Celery/RQ/worker queue
* **파일 저장:** S3 호환 스토리지
* **임베딩 저장:** pgvector
* **AI 오케스트레이션:** 별도 service layer

### 상세 구현 모듈

#### 1. Source Ingestion Module

역할:

* 텍스트, PDF, 링크 수집
* 전처리
* 문단 분리
* 메타데이터 생성

구현 포인트:

* 긴 문서를 paragraph chunk로 자름
* chunk별 길이 제한
* 링크는 본문 추출 후 정리
* PDF는 텍스트 추출 실패 시 대체 처리 고려

저장 테이블:

* sources
* source_chunks

#### 2. Card Generation Pipeline

흐름:

1. source를 chunk 단위로 분해
2. chunk마다 개념 후보 추출
3. 개념을 클러스터링
4. 개념별 카드 생성
5. 카드 후처리

   * 중복 제거
   * 질문 길이 검사
   * answer 적정 길이 검사
6. 결과 저장

저장 테이블:

* concepts
* cards
* generation_jobs

핵심:

* 카드 생성은 비동기로 처리

## 8. 제약 조건

### 기술 제약

1. AI 응답 품질은 완벽하지 않다
   따라서 검증/후처리/수정 UX가 반드시 필요하다.

2. 실시간 AI 호출은 비용과 지연시간을 수반한다
   모든 기능을 실시간 생성으로 설계하면 안 된다.

3. PDF/링크 추출은 원본 품질에 따라 실패할 수 있다
   예외 처리와 fallback이 필요하다.

4. 멀티플랫폼 동기화는 복잡하다
   오프라인 우선 전략을 쓸 경우 sync 충돌 처리 설계가 중요하다.

### 제품 제약

1. 초기에는 모든 과목/도메인을 완벽히 지원할 수 없다
   범용 구조를 갖추되, 실제 고도화는 특정 학습군부터 시작하는 것이 낫다.

2. 너무 많은 AI 개입은 사용성을 해친다
   유용한 순간에만 나타나야 한다.

3. 너무 캐주얼해 보이면 파워 유저가 신뢰하지 않는다
   강력한 도구의 인상을 잃지 않아야 한다.

### 비즈니스 제약

1. 무료 플랜의 AI 사용량은 제한 필요
2. 고급 기능은 분명히 유료 가치가 있어야 함
3. 초반에는 acquisition보다 retention이 더 중요
